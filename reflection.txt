## Week 6 Reflection

1. What is the key difference between calling an LLM once and using an agent? Use an example from today's lab.

The key difference is that a single LLM call only produces one response from the prompt it is given, while an agent can reason through a task, choose tools, use those tools, and continue working until the request is completed. In this lab, a normal chat endpoint could answer a question about movies, but the agent could actually manage the tracker. For example, if the user says they finished a movie, want to rate it, and also want to know what they are currently watching, the agent needs to call multiple tools in sequence instead of answering in one step.

2. The agent receives tool results back as "user" role messages. Why does this work? What does it tell you about how LLMs handle context?

This works because the model uses the full conversation history as context and continues reasoning from whatever structured information it is given. Once a tool result is inserted back into the message list, the model can read that result and decide what to do next. This shows that LLMs are flexible about context as long as the information is clearly formatted and attached to the right tool-use step. The important thing is not the human label itself, but that the model receives the tool output in a consistent conversational structure.

3. What would happen if your tool descriptions were vague or incorrect? Give a specific example of how bad descriptions could cause wrong behavior.

If tool descriptions were vague or wrong, the agent could pick the wrong tool or use the right tool in the wrong way. For example, if the description for `update_movie_status` did not clearly mention that it can also store a rating, the agent might mark a movie as watched but fail to save the user's star rating. If the description for `get_movies` did not explain that it supports filtering by status, the agent might fetch the entire library when the user only asked what they are currently watching. Clear tool descriptions are important because the model depends on them to make good decisions.

4. You now use Claude Code every day. Describe its behavior in terms of what you learned today: what tools does it likely have? How is the agent loop running?

Claude Code likely has access to tools for reading files, editing code, running shell commands, searching a codebase, and sometimes using browser or external integrations. Its behavior feels very similar to the agent loop from this lab: it interprets the request, decides whether a tool is needed, calls the tool, reads the result, and keeps going until it can produce a final answer. The difference is that Claude Code has a much richer tool environment than my movie tracker agent. This lab made it much easier for me to understand how that tool-using workflow actually works.

5. What could go wrong if an agent had the ability to DELETE books and there was no human-in-the-loop check?

Without a human-in-the-loop check, the agent could delete the wrong item because of ambiguity or a mistaken interpretation of the request. For example, if a user says "remove the Orwell book," the agent might delete the wrong title if there are multiple matching books or incomplete data. This would be dangerous in any app that manages important records. A safer design would include confirmation before destructive actions, better matching logic, or a soft-delete system so mistakes can be recovered.
